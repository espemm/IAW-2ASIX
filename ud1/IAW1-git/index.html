Introducción a Git y GitHub¶

En aquest taller d’introducció a git i GitHub repasarem/aprendrem les ordres bàsiques per començar a treballar amb repositoris de manera local i remota.

Material basat en el taller d’introducció a Git i GitHub de José Juan Sánchez i al llibre “Pro Git” de Scott Chacon and Ben Straub Pots accedir a aquest taller al següent enllaç:

  • Pro Git book
  • Taller d´introducció a git i GitHub

Git / GitHub

Git és un programari de control de versions dissenyat per Linus Torvalds. Ens ajuda a portar el registre dels canvis dels fitxers d’un repositori (directori).

GitHub és un servei que ens permet guardar repositoris Git a un servidor remot.

GIT¶

Instal·lació de git¶

Linux

$ apt install git
Windows Descarregar des de la web oficial: http://git-scm.com/downloads.

Configuració de git.¶

Configurem el nom i l’email que apareixeran als commits que fem sobre els repositoris.

git config --global user.name "Nom Cognoms"
git config --global user.email us@domini

Comprovar si s’han aplicat els canvis:

git config --list

Estats d’un fitxer en git¶

Un fitxer pot estar en algun dels estats següents:

  • Sense seguiment (untracked)
  • Preparat (staged)
  • Modificat (modified)
  • Confirmat (commited)

Ús bàsic de git¶

  • Crear un directori

    $ mkdir curso-de-git
$ cd curso-de-git

  • Inicialitzem el repositori:

    $ git init
Inicializado repositorio Git vacío en ...

    Note

    Se ha creado el directorio .git. El fichero de configuración del repositorio en .git/config

  • Afegir fitxers

    Afegim un nou fitxer al repositori:

    echo "Estem fent una prova">exemple.txt
git add ejemplo.txt

    I confirmem els canvis:

    git commit -m "He creat el fitxer exemple.txt"

  • Modificar un fitxer

    Al modificar un fichero, podemos confirmar el cambio pero tenemos que usar la opción a:

    nano exemple.txt
git commit -am "He modificat el fitxer exemple.txt"

  • Estat del repositori

    git status

  • Llistat de commits:

    git log

  • Altres operacions

    • Canviar el nom al fitxer:

      git mv exemple.txt exemple2.txt
git commit -am "He canviat el nom del fitxer"

    • Eliminar un fitxer

      git rm ejemplo2.txt
git commit -am "He borrado el fichero ejemplo2"

    • Tornant a versions anteriors

      Podem tornar a l’estat anterior. Si volem tornar a un commit anterior sense perdre les modificacions posteriors:

      git checkout <id del commit>

    • Per a tornar a l’última versió:

      git switch -

    • Si vull eliminar els 3 últims commits:

      git reset --hard HEAD~3

GitHub¶

GitHub és un servei web que permet tenir repositoris git remots en un servidor.

Ens permet tenir sincronitzat el nostre repositori local, amb un repositori remot.

Per connectar-nos al servidor remot de GitHub utilitzarem una connexió SSH o TOKEN.

Necessitem que en la connexió SSH no ens demane la contrasenya, per tant cal configurar en el servei la nostra clau pública (apartat “SSH keys” del teu perfil en GitHub).

Creació de repositoris¶

El més senzill és crear un repositori a GitHub i després clonar-lo en el teu ordinador.

Utilitzarem la URL ssh per a realitzar la clonació.

Utilitzarem la URL https per a clonar repositoris que no són nostres. En aquests repositoris no podem fer modificacions.

Per a clonar-lo:

```bash
git clone git@github.com:xxxxxxx/xxxxxxx.git
```

Per a comprovar que tens configurat el repositori usant la url SSH pots vore el fitxer de configuració en .git/config.

Sincronitzant repositori local i remot¶

Per pujar els canvis des del nostre repositori local al remot fem servir:

git push

Per descarregar els canvis des del nostre repositori remot al local fem servir:

git pull

Afegint un repositori remot

Crearem un repositori local, però el nom de la branca ha de ser main:

git init -b main

Si ja tenim un repositori, podem canviar el nom de la branca:

git branch -M main

Creem un repositori buit (Sense fitxer README.md) a GitHub. Afegim el repositori remot i sincronitzem:

git remote add origin git@github.com:espemm/prova.git
git push -u origin main

Treballem amb rames i unions¶

Rames

Una branca representa una línia independent de desenvolupament, és com crear una nova àrea de treball que tindrà el seu historial propi de commits. Per llistar les branques locals executa:

git branch
* main

La branca en què estàs treballant actualment s’assenyala amb un asterisc*. GitHub crida a la branca principal main. En repositoris antics s’anomena màster. A la branca principal és on sol trobar-se la darrera versió del repositori

Ramificar

Creem una nova branca, per fer canvis al repositori (sense tocar l’estat de la branca principal). Si els canvis són correctes, hem de fusionar la branca creada amb la branca principal. Per crear una nova branca:

git branch [rama]

Per a passar a la nova rama utilitza el comandament:

git checkout [rama]

La nova branca té els mateixos fitxers que a la branca principal. Tots els canvis que facis als fitxers en aquesta branca no es veuran a la branca principal. Truc: amb l’ordre git checkout -b [branca] es crea una nova branca i t’hi posiciones. Amb l’ordre git branch -v es veu el darrer commit de cada branca. Comproveu que coincideixen l’últim commit a les dues branques. Truc: Pots fer servir una extensió del teu shell (bash, zsh,. . . ) perquè et mostri en el prompt la branca en què estàs treballant.

Sincronitzar rames amb el repositori remot

Les branques no es creen automàticament a GitHub, cal fer un push per crear-les en remot. Nota: ‘origin’ és el nom del dipòsit remot.

git push origin [rama]

Eliminar i fusionar rames

És força freqüent crear una branca, fer els canvis que siguin necessaris, unir-la a una branca principal i després eliminar la branca que s’havia creat.

git branch -d [rama]

Quan has treballat en una branca, el normal és voler incorporar els canvis a la branca principal. Per **unir/fusionar una branca a la principal:

git checkout main
git merge [rama]

Conflictes¶

Quan només s’han afegit o eliminat fitxers en una branca, és fàcil unir-la a la principal. Quan es fan modificacions en fitxers, incloent canvis en els noms dels fitxers, git detecta aquests canvis i els adapta automàticament, però de vegades sorgeixen conflictes. Els conflictes apareixen quan la mateixa part del codi s’ha modificat en dues branques diferents.

cat prova.txt
<<<<<<< HEAD
Hola com estàs
=======
hola que tal
>>>>>>> nuevo

Pull Request¶

Què és un Pull Request?

  • Un Pull Request és una petició que es fa al propietari d’un repositori original perquè aquest darrer incorpori els canvis que se suggereixen.
  • Els Pull Requests han esdevingut la forma més habitual de col·laborar en projectes de programari lliure.
  • Com col·laborar en un projecte de programari lliure? Què és un Pull Request?

Passos per fer un PR

  • L’usuari ha de fer un fork del dipòsit al qual vol contribuir.
  • Clonar el repositori.
  • Crear una nova branca, on realitzarem els canvis que posteriorment proposarem com a canvis.
  • Pugem els canvis al nostre repositori, creant una branca al repositori remot.
  • Crear el Pull Request des de la pàgina de GitHub.
  • El propietari del repositori a què hem realitzat el PR haurà de validar el canvi. I decideix si ho accepta, fent una unió amb la branca principal, o ho rebutja.

Sincronització del nostre fork amb el repositori original

En fer el fork del repositori original fem una còpia completa en un moment determinat, però aquest repositori pot anar canviant. Aquests canvis no es veuran reflectits al nostre fork de forma automàtica.

Abans de proposar un canvi (fer un Pull Request), tenir el nostre fork tan actualitzat com sigui possible perquè no tinguem problemes que produeixin conflictes.

Ens posicionem a la branca principal del fork. Afegim un repositori remot apuntant al repositori original (upstream). Busquem els possibles canvis que ha tingut el repositori original:

git fetch upstream
fusionem els canvis del repositori original que estan a la branca principal.

git merge upstream/main
